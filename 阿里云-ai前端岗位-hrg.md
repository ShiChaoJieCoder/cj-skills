# 自我介绍
您好，我是一名前端工程师。**简历上我刻意把主线写在高性能渲染与复杂交互**，比如移动端长列表、Web 3D——虚拟列表、预加载与解码策略、设备分级与 LOD 等，把「能打开」和「够流畅」做成可验收指标。

**在公司我其实做过接近 AI 产品的智能问答落地**（例如数字展厅场景）：会话复用、轮询与占位态、追问链路和可信展示，解决「等不及」「不信」「追问断」等问题。之所以没有单独在简历里包装成「AI 产品」模块，是因为当时链路相对偏早期范式（和今天常见的流式输出、工具调用、Agent 编排、多模型协同相比，技术代际会显得旧一些），我担心写成大块「AI 经历」反而让面试官误判我的技术栈还停留在上一代，所以面试里我愿意主动把这页补齐：**我有真实问答类产品的前端闭环经验，同时我也在自学和演练新一代 AI 工程方法**，两边我能讲清楚差异和迁移点。

和阿里云智能 AI 前端岗位特别契合的一点，是我 **系统性自学 AI 工程**，而不是停留在会用工具：我有一套自己的 **AI 开发体系**——用 **Spec-Kit + Superpowers** 把需求与边界写清楚，再用 TDD、评审和验证闭环管住交付质量，让人定义问题、AI 加速执行、人对结果签字。

在 **Agent** 上我不是只背概念，而是按可落地的模块拆解：**Agent Loop（感知—规划—执行—复盘）**、**Tool System（工具注册与调用约束）**、**Context Engineering（上下文裁剪与注入策略）**、**Memory（短期会话与长期记忆边界）**、**Multi-Agent（分工与协作编排）**，以及 **Harness / 评测与回归（把「偶发能跑」变成「可评估、可迭代」）**。我认为 AI 前端的长期价值，就是把模型能力与云上资源串成 **用户可感知、业务可度量** 的闭环——对齐「客户第一」：技术指标最终要能翻译成等待感、可信度、转化等行为结果。协作上我习惯 **把边界和需求写清楚**（Spec / PRD / 验收标准），减少猜来猜去，也更贴近「因为信任，所以简单」。迭代上我认同 **「今天最好的表现是明天最低的要求」**：上一版达标只会被我当成下一版的基线，通过埋点与复盘持续抬高标准。

# 你过往最具挑战的项目是什么？你承担了什么角色？

# 你如何应对高压与不确定性的项目？
高压下最怕的不是忙，而是 **目标发散**。我会先做三件事：第一，和业务对齐北极星指标（用户能感知的结果优先），对应「客户第一」；第二，把未知拆成可验证假设，快速试错，对应「拥抱变化」但不蛮干；第三，设里程碑和同步机制，进度和风险透明，减少互相猜疑，对应「因为信任，所以简单」。  
例如 3D 项目一开始既有崩溃又有卡顿，我把问题分成「加载链路」和「渲染链路」并行验证，每周有可演示进展，团队心理压力也会小很多。

# 研究路线怎么确定的
我通常按“现象 -> 指标 -> 假设 -> 实验 -> 回归”来定路线。先确认瓶颈是 CPU、内存、网络还是主线程阻塞，再设计最小实验验证。验证有效才放大实施，实施后做回归和监控，确保不是局部优化、全局退化。路线不是一次定死，而是持续根据数据调整。

# 遇到了什么困难，如何克服的
我遇到过的最大困难之一是 H5 长列表在低端机严重卡顿：一次性渲染导致 DOM 过多、图片解码挤占主线程、滚动掉帧明显。仅做虚拟列表仍会在快速滑动时被大图并发加载拖慢。  
我把问题拆成三线治理：虚拟列表控制 DOM 数量，`IntersectionObserver` + `rootMargin` 做预加载，`decoding="async"` 和定高骨架控制解码阻塞与 CLS。最后把性能从“可用边缘”拉到“稳定可用”，FPS 提升到 55+ 量级，内存明显下降。

# 遇到了什么反尝试的问题，如何解决的
我做过一个反尝试：在 3D 场景里先尝试“统一降画质”来保性能，结果高端设备体验明显变差，业务方反馈很差。  
我及时复盘后改成“设备分级 + LOD + Worker 解码”的组合策略：低端优先保稳定，高端尽量保细节，中端平衡。这个反尝试让我认识到，性能优化不是单目标最优，而是分层策略下的整体最优。

# 所负责的项目中哪个最有挑战或有成就感？（从业务角度具体展开）
最有成就感的是数字展厅 AI 问答体验优化。初期核心问题是用户「等不到反馈」、答案可信感弱、追问链路断——都是 **客户侧真实痛点**，不是后端通了就行。

# 如何向不懂技术的人解释云计算？
云计算可以理解成“把原本要自己买、自己维护的服务器和软件能力，变成像水电一样按需使用的服务”。  
你不用先投入很大成本搭机房，只要按业务需要开通资源，流量大就扩容，流量小就缩容，省钱也更灵活。

# 别人对你的评价怎么样？最负面评价是什么？
正面评价一般是靠谱、抗压、推进力强，复杂问题不会往外甩——这和主人翁意识（非我莫属）比较接近。  
负面有时会听到「太较真」。我的复盘是：**较真要对准客户价值和关键路径**，而不是在无差别的细节上消耗协作信任；所以现在我会先锁优先级与时间盒，把「必须先赢的一次交付」说清楚，再深度打磨——既不糊弄客户，也不让同事难受。

# 举一个跨部门合作的案例
在数字展厅 AI 问答优化中，我和产品、算法、后端高频协作：产品要转化、算法要质量、后端要稳定——表面上 KPI 不同，但 **共同客户是终端用户**。我先把各方语言翻译成同一套可观测指标（等待时长、有效回答率、追问转化等），再落成统一会话、状态机和兜底策略；埋点数据 **双向透明** 给算法迭代，减少「我觉得」扯皮。这和「因为信任，所以简单」一致：**信息对齐了，协作成本就会下降**。最终体验和转化都有正向结果。

# 团队起冲突，怎么解决？
我会先把冲突从「立场」转成 **共同目标和事实**：我们对齐的是不是同一个客户问题、同一套优先级（客户第一）。然后把分歧拆成可验证假设，用数据或小实验决策，避免无限辩论。  
若短期无法统一，我会推动 **低风险试点 + 明确复盘节点**，节奏上不耽误交付，态度上对同事保持透明和尊重——信任是靠一次次说到做到建立的（因为信任，所以简单）。

# 你与团队起冲突，怎么解决？
如果我和团队有分歧，我先自检两点：**信息是否对齐**、**表达是否让人误会**。然后把假设、证据和风险 **摊开来讲**，欢迎同事挑战我的盲区，而不是暗暗较劲。  
我坚持对事不对人；需要拍板时服从决策，但会把复盘写清楚——这也是「今天最好的表现是明天最低的要求」：冲突不可怕，可怕的是重复同样的误会。

# 面对利益冲突时怎么解决？
我会按「客户第一，员工第二，股东第三」来排序：**不能牺牲终端客户的信任换短期指标**，也不能为了局部省事伤害公司长期口碑；合规与诚信是底线，没有讨价还价空间。  
落地上我会 **把利弊和决策依据透明化**，该升级就升级，留下记录，避免灰色地带——长期看，简单透明的决策反而成本最低（因为信任，所以简单）。

# 你对阿里云所在行业和业务趋势的看法
我认为趋势是“模型能力普及化，行业场景精细化，云上交付一体化”。未来竞争点不只是模型参数规模，而是场景落地效率、成本和体验。  
以秒悟这类产品为例，市场正在从“辅助编码”走向“描述即生产、快速上线验证”。这对前端提出更高要求：不仅要做好界面，还要做好多模型协同下的状态管理、可解释反馈和全链路可观测。  
对前端而言，机会在于把 AI 能力产品化：更低延迟感、更高可信度、更好的多端一致性和更强可观测性。

# 公司的 OKR 指标是什么？
我理解 OKR 一般会围绕业务增长、用户体验和交付效率三类目标。对前端岗位，我会主动把目标拆到可量化指标，比如页面性能、稳定性、关键转化漏斗和迭代周期。  
我希望做的是“把业务 OKR 映射成工程动作”，保证每个技术动作都能对应业务价值。

# 每个 O 完成的评价标准是什么？
我会用“结果 + 质量 + 可持续性”来衡量：结果是否达成核心指标，质量是否稳定可复用，可持续性上是否沉淀了机制和规范。  
例如优化项目不只看一次上线数据，还要看回归稳定性和后续团队复用效果。

# 反问环节：组织架构、试用期、转正标准、入职时间
我会这样反问：秒悟前端团队在应用生成链路里主要负责哪些核心模块（如生成态交互、发布链路、可观测）？试用期最核心的 1-2 个目标是什么，转正评估更看重哪些结果？  
如果双方确认匹配，我可以配合业务节奏尽快安排入职时间。我个人是非常希望加入这个团队的，也愿意在前期承担更高强度的攻坚任务。

# 技术更新快，你如何保持敏感度和先进性？
我默认「唯一不变的是变化」，所以会 **并行做输入与输出**：跟进官方与一线实践，同时每季度做 1～2 个可验证的小实验，而不是囤积教程。  
方法上用固定框架控风险：边界清楚 → 小步实现 → 测试与证据收尾；昨天有效的方法我会复盘是否仍是明天的下限（今天最好的表现是明天最低的要求）。只有跑出来指标变好，我才主张推广到团队，避免追新概念给客户和同事埋坑（客户第一）。

# 最近在关注哪些新技术？
我主要关注三块：AI 前端工程化（多轮会话、流式响应体验、Agent UI 形态）、性能与稳定性（Web Vitals、资源调度、监控告警）、3D/多媒体性能优化（渲染管线与跨端适配）。  
结合秒悟体验，我也在重点关注多模型协同产品中的前端问题，比如生成过程可视化、错误修复反馈、从“可生成”到“可维护”的体验设计。  
在 Agent 方向上，我正在持续学习和拆解 Agent Loop、Tool 调度、上下文工程、记忆机制、多 Agent 协作与评测体系，目标是把“能跑 Demo”提升到“可评估、可迭代、可上线”。
关注重点不是“新名词”，而是哪些技术能真实提升用户体验和业务结果。

# 入职后怎么快速适应公司节奏和业务？
我会用「30-60-90 天」：30 天吃透业务链路、关键指标和协作接口，多问少说猜；60 天交付 **一个可量化的小胜仗**（例如生成态等待体验或转化链路），证明我能创造价值；90 天沉淀可复用组件或规范，抬高团队下限（今天最好的表现是明天最低的要求）。  
全程主动对齐上级预期，优先级透明。我的风格是 **该扛的责任不往外推（此时此刻，非我莫属）**，但也尽量用文档和同步让协作简单。

# 场景题：算法同学优先级更高，导致项目瓶颈，怎么解决？
核心是 **客户要不要在这条需求上等**：我会和产品先把业务优先级对齐（客户第一），避免团队私下猜各自最重要的事。再和算法拆分：**哪些必须模型升级，哪些前端可以先兜住体验**——例如交互闭环、会话与占位、兜底与埋点先行，先让用户「有感」，算法到位后再换高质量答案。  
资源冲突时我会 **把进度和风险透明同步**，让大家在同一事实面上协作（因为信任，所以简单），而不是前端单方面抱怨或硬扛延期。

# 你认为 AI 研发和传统开发的区别
传统开发更强调确定性流程和规则实现，AI 研发更强调概率性结果和持续迭代。  
能力要求上，AI 场景除了工程能力，还要求你有实验设计、效果评估和跨角色协作能力；工作内容上也更需要快速试错和数据闭环。  
我自己的理解是：传统开发偏“功能正确性”，AI 研发还要管理“上下文正确性”和“工具调用正确性”。所以我会额外关注 Context Engineering、Memory 设计和 Harness 验证，确保系统稳定输出而不是偶发可用。

# AI 的出现会让你焦虑吗？
不会回避焦虑，但我更把它当成升级信号。AI 会替代重复性执行，但也会放大“问题定义、系统设计、业务判断”的价值。  
我会持续学习并把 AI 融入工作流，让自己成为“会用 AI 提升交付质量和速度”的工程师，而不是被动等待变化。  
简历上 AI 标签不重，一部分是我 **有意取舍**：早期问答类产品的前端实现方式和当下爆款范式不完全一致，我不想用标题赢眼球、用细节却对不上预期。另一部分我会用 **持续投入** 补足：系统学习框架、沉淀开发体系、做小实验验证。把不确定性转成可证明的进步，是我一贯的应对方式。

# 简历上 AI 背景不重，你如何证明你能胜任 AI 前端岗位？
我会从三个层面来证明，落脚点都在 **客户可感知的结果**。第一是可迁移能力：高性能与复杂交互，本质是高压下的拆解、指标治理与稳定交付——AI 产品同样需要你扛不确定性（拥抱变化）。  
第二是真实实践：**公司里做过智能问答前端闭环**，简历写法不等于经历为零；我愿意在面试里说清楚业务贡献，并补足对流式、工具、Agent 等新链路的理解与实验（Spec-Kit / Superpowers 与各 Agent 模块的持续拆解）。  
第三是担当与结果：**入职后我会主动认领一条链路打到量化指标（此时此刻，非我莫属）**，用数据和复盘说话，而不是只强调「我会用工具」。这既对客户公平，也对团队简单透明。

# 遇到过的最大的困难和挫折，如何克服的？（体现“皮实”）
我遇到过最大的挫折，是一个 3D 场景项目临上线前一周问题集中爆发：低端安卓频繁崩溃，高端机又反馈画质下降，发布节点已定，压力集中在同一周。团队情绪紧张，我们前期还走过弯路（一刀切降画质），等于 **用一个痛点换另一个痛点**，这在客户视角并不成立（客户第一）。  
后来我强迫自己回到事实面：把问题拆成「能不能打开」和「能不能流畅」，上设备分级 + LOD + Worker 解码，按日回归、指标透明同步给业务方，让大家看见曲线在变好——协作简单了，信任也容易回来（因为信任，所以简单）。最终按节点上线。  
这件事给我的标签是：**顺风局谁都能写代码，真正难的是逆风里还能把团队往前推、把结果扛下来（此时此刻，非我莫属）**。皮实不是歌颂熬夜，而是压力下仍然冷静、乐观、可依赖。
