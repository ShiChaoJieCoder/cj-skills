# 自我介绍

您好，我是一名前端工程师。**简历上我把主线押在「高性能渲染 + 复杂交互」**，平时在项目里多是 **出方案、推着落地** 的那种角色。

一块是 **移动端 H5 长列表**：虚拟列表压住 DOM、懒加载和预加载、`decoding="async"` 配上骨架屏，减轻解码和 CLS，低端机上滚动顺不顺、内存扛不扛得住我都会盯；FPS 大致能稳在 **55+** 那一档，事后也用数据复盘，不靠嘴上说「感觉快了」。另一块是 **Web 3D**：设备分级、LOD、Worker 里解码大图，免得一张大图把主线程拖死；也折腾过崩溃和「要画质还是要帧率」的拉扯，上线前那种高压周算见识过。说白了，**「能打开」和「够流畅」** 我都尽量落成 **能验收的数字**，不停在体感。

**公司里也做过靠近 AI 产品的智能问答**，比如数字展厅那条线：会话怎么复用、轮询和占位怎么让用户别干等、追问怎么不断档、答案旁边怎么让人敢信——对准的就是「等不及」「不信」「追问断」。简历里我没单独铺一大块「AI 产品」，主要是当时那套实现和现在常说的流式、工具调用、多 Agent 编排不那么同一个代际，我 **不想靠标题唬人、细节又对不上**；面试里我愿意把这页补全：**问答类产品的前端闭环我是真做过**，新一代 AI 工程我也在自学和演练，**两套我能讲清楚差在哪儿、怎么往上迁**。

**平时写代码怎么配合 AI：**我按自己总结的 **Spec-Kit + Superpowers** 那一套来（仓库里有篇 `spec-kit-superpowers-开发方式总结.md`）。翻译成大白话就是：**先把做什么、验收啥写死**，再让 AI 帮着写实现；执行侧尽量 **先测后写、审查和验证过关**，**人对结果签字**，不把「偶尔能跑」当上线标准。

说到 **智能体这类东西**，我不会只会背名词，脑子里会想 **真要上线得拆开哪些事**：听明白需求 → 拆几步去做 → 做完回头看有没有跑偏；**什么时候能手伸到外面调接口、查数据**，规矩和报错怎么处理；对话长了 **上下文怎么裁剪**；**这轮会话记多少、长期记忆边界在哪**；假如多个助手分工，**谁偏界面谁偏推理** 也要编排顺；最后还得有 **测试和回归**，不能十次里只有一次运气好。**落到前端**，就是把模型和云上能力收成用户 **等得起、看得懂、愿意接着用** 的面儿上的体验；协作上我习惯 **早点把边界和需求说明白**，大家少猜；迭代上 **上一版达标只当下版的起跑线**，埋点和复盘把标准往上抬。

**Multi-Agent（分工与协作编排）**，以及 **Harness / 评测与回归（把「偶发能跑」变成「可评估、可迭代」）**

我离职的主要原因是职业发展方向更加聚焦在 AI 前端和云智能场景，但原岗位在这方面的业务占比有限。不是对原团队不认可，而是希望在更贴近 AI 产品核心链路的环境里，把「模型能力到用户体验」这段路做得更深。  
从价值观上说，我特别认同「让天下没有难做的生意」需要通过 **云 + AI 降低交付门槛**：秒悟这类「描述即生产、快速上线验证」的方向，本质上是在帮客户更快拿到结果，这和「客户第一」高度一致。行业里 AI 产品形态变化非常快，我也希望自己守在 **变化最快的核心战场**，而不是做过一轮问答之后就停在舒适区——这点和我对「唯一不变的是变化」的理解是一致的。我对这个岗位的意愿非常明确，也愿意长期投入。

# 个人优缺点

优点第一是抗压和韧性强，面对不确定问题不会慌，会拆解到可执行粒度——AI 场景本身多变，这和「拥抱变化」是匹配的。第二是 **客户与业务视角的指标意识**：做优化会把「用户等待、可信感知、转化」和「性能数字」一起讲清楚，避免自嗨型优化。第三是协作上追求 **简单透明**：主动同步风险、写清边界与假设，减少信息不对称，我觉得协作成本越低，信任越容易建立（因为信任，所以简单）。  
不足是有时前期会把方案想得过于完整，节奏偏稳。现在我刻意练习 **小步验证、快速对齐**：先把最小可用版本跑通，再用数据决定纵深投入，既跟上变化，也守住交付节奏。

# 在团队管理上，遇到过什么困难？

我刻意挑 **Web 3D 临上线** 这段讲，因为难点主要在 **带团队在高压下不失控**，而不只是我自己写代码。当时我是 **前端负责人** 兼 **SM**，前端 **3 正编 + 3 外包**：业务侧 **节点钉死、诉求对立**（低端要稳、高端要观感），团队里又因为 **错误的一刀切策略** 出现 **互相埋怨和执行摇摆**——这是典型的 **管理场景**：**目标不清 + 责任边界不清 + 情绪上来**，技术再好也会被拖垮。

我当时从 **管理动作** 上做了几件事：

**一是「对齐目标、管住预期」。** 主动拉上业务、测试做 **短对焦**：承认前一版策略有问题，明确接下来 **唯一认可的路线**（分级 + LOD + Worker），把 **Must 验收** 写成大家都能复述的两句话，避免各说各话；同时 **挡掉模糊施压**——群里再催也回到「对照验收表」说话。

**二是「分工授权、谁拍板谁执行」。** 我把前端拆成 **高危核心** 与 **可并行外围**：核心链路 **授权给正编owner**，我保留 **关键决策和评审权**；外包只做 **边界清晰、可独立验收** 的包（多机型复测、参数表、演示走查、埋点对齐），每条任务 **Definition of Done 写死 + 截止时间 + 必过 CR**——这是 **授权**，不是甩锅；避免「谁空谁上」导致返工。

**三是「过程可视、化解冲突」。** 正编外包摩擦本质是 **信息不对称**：我用 **每日简报** 把「机型 × 指标」变化摊开，让贡献 **看得见**；站会我当 SM **只处理阻塞和优先级**，有争议 **当场拉到后端/产品桌上拍**，**不让矛盾在小组里隔夜发酵**。

**四是「风险升级不硬扛」。** 涉及路线或资源争议，我 **带着时间线和影响面** 找能拍板的人一次定夺，避免团队用加班掩盖管理缺口。

**结果**：**按节点上线**，业务对演示与稳定性认可；团队复盘时也认可 **后面几周节奏是稳的**。我自己把它收成一句话：**管理能力体现在——危机里能不能快速对齐目标、拆清责任、让进度透明、把冲突收成决策**；技术方案是我作为负责人要兜底的，但 **没有这几条管理动作，mixed team 很难一起扛住交付**。

# 管理五个人的团队和管理五十个人的团队有什么区别？

我的理解是：**人数上去以后，靠「队长一个人盯全局」就不够用了，要靠机制和分层。**

**五个人左右**，很多时候还能 **全员同一套上下文**：大事小事口头对齐也快，你能 **直接看到每个人的进度和质量**，必要的时候 **自己下场抠细节** 也能 cover；流程可以轻一点，**信任和默契** 占比高。

**五十人量级**，个人精力 **不可能覆盖到每一个人**，一定要 **拆成小团队 + 二层负责人**，管理者的工作重心会从「亲自解题」更多转向 **定方向、定标准、定节奏**：目标怎么层层对齐、质量底线怎么统一、跨团队依赖怎么制度化同步、风险怎么升级；信息传递也不能只靠走廊聊天，而要依赖 **例会节奏、文档与数据看板、复盘机制**，不然一定会失真或断层。

**不变的是**：目标要清楚、责任要落到人、承诺要兑现。**变化的是**：小团队更像 **「一起扛」**，大团队更像 **「体系扛」**——考验你有没有 **可复制的方法论** 和 **能替你扛一线的中坚力量**。

我目前带的是 **个位数到十来人** 的规模，还在积累「盯人」与「盯事」的平衡；如果未来有机会带更大盘子，我会刻意多练 **拆授权、建机制、培养 TL** 这几件事，而不是单纯把自己拉长。

简历里我主线是 **长列表、Web 3D** 的性能与交互治理，加上 **展厅智能问答** 的前端闭环——等人、追问、可信展示这几件事——正好对上 AI 前端要的 **性能底盘 + 面上体验 + 能扛上线压力**；平时也是方案落地一路推到指标验收，不是纯接需求。在这个基础上，我更多是 **对着接下来几年想长成什么样** 来看这个机会的。**接下来一两年**，我希望自己能稳稳地 **独立扛起 AI 相关产品里的前端核心模块**——既要抠交互和性能，也要习惯跟算法、后端一块儿把东西推到上线，结果是能说得清的，而不是「界面做完了就行」。**再往中期看**，我想沉淀出一套在 AI 场景里还能复用的打法：性能预算、体验指标、可信交互、数据怎么闭环，后面带团队时能用得上。**长线**，我愿意在云上的 AI 产品里参与 **真有业务分量的技术决策**，始终把「用户这边是不是更简单、更可信」当成判断题的题干。

所以选这个岗，核心还是它能把我 **往我想去的方向再推一把**，在核心业务里把「能用」往「好用、敢长期用」再去磨一层。

# 遇到过的最大的困难和挫折，如何克服的？（两个困难）

高压下最怕的不是忙，而是 **目标发散**。

我印象最深的一次，是一个 **Web 3D 场景临上线前大概一周彻底失控**：低端安卓 **点开就崩、崩完再点还是崩**，日志里一看内存和主线程全在报警；高端机那边业务又顶着骂过来说 **细节糊成一团**，演示根本没法看。发布节点是死的，群里从早到晚在催，团队里有人已经开始说「要不算了延期」——那时候气氛其实很压抑。更糟的是我们前两天还走过弯路：**为了救命一刀切全局降画质**，结果低端没彻底救回来，高端先被得罪了，等于 **左边漏风右边也漏**，我自己心里也知道这是在赌运气。

我当时强迫自己 **先别跟着情绪跑**，回到桌面上把事情摊开：第一步，承认「降画质一条杠」是错的，跟业务和测试 **当面说清楚我们先 rollback 策略**，先把预期对齐，别让猜疑扩大。第二步，把故障 **劈成两条线并行**：一条只管 **「能不能稳定打开」**（崩溃、加载、内存峰值），另一条只管 **「打开之后能不能保住观感」**（帧率、清晰度、LOD 切换是否突兀），各有各的验收口径，不再混成一团账。第三步，方案上改成 **设备分级 + LOD + Worker 里解码大图**，低端默认吃「能跑、少崩」的配置，高端单独拉高上限，中端取平衡——每一档都能在真机上 **对照表格说清楚改了啥**。第四步， **按天回归**：我每天固定产出一份「哪些机型、哪些指标比昨天好」，同步给业务和产品，让大家看见曲线是真的在往上走，而不是口头安慰。中间几天确实很熬人，但节奏稳住以后，团队至少知道 **每天都在逼近节点**，不是原地打转。

最后结果是：**按原定节点上线了**——低端崩溃率压到可接受范围，演示机型上流畅度和观感业务也能签字；回头看，那段时间对我来说 **皮实** 不是比谁睡得少，而是 **场面最难看的时候还能把脸埋进事实里、一步一步把坑填平**，让别人觉得你 **靠得住**。

**另一类印象很深的挫折，是数字展厅的 AI 问答。** 前期同样 **很难对外演示**：用户一问就 **长时间没动静**，不知道系统在没在干活，现场有人直接说「像坏的」；答案出来了又 **说不清依据**，大家不敢信；想 **追问** 常常 **断在半路上**，业务压力大的时候问过要不要先撤掉这个功能。算法和后端也在扯延迟和质量，前端夹在中间，典型的 **「接口通了，场面仍兜不住」**。

我还是 **先把情绪和口径压住**：跟产品、业务把痛点钉成三件事——**空等、不信、追问断**，每一条都要有 **用户能感知、我们能量化** 的验收说法，而不是互相埋怨。然后 **一层层拆**：**感知层** 先做「先给反馈、内容再慢慢回填」，别让界面一片空白把人吓跑；**质量层** 做会话复用，并把参考资料或溯源信息展示出来，让人觉得「有据可查」；**业务层** 把追问状态和边界兜全，断档要有兜底引导，而不是 silent fail。**埋点和漏斗** 同步铺开，把等待时长、有效回答、追问转化这些数据摊开给算法和后端迭代，减少「我觉得慢」这种口水仗。几轮迭代下来，**等待焦虑下去了、可信感稳住了、追问和咨询转化也能看出来在变好**，业务才重新敢拿去对外讲。

**成就感我会分开说**：**3D 那场** 证明的是 **崩盘边缘还能守住节点、团队觉得你靠得住**；**展厅这场** 证明的是 **前端能把 AI 能力收成用户真的愿意用、业务愿意背书的体验**，改动最终能说到 **等待、信任、转化** 这些结果上，而不只是「接口联调成功」。若面试官接着问「你最大的成就是什么」，**想听韧性和工程兜底就讲 3D**；**想听用户和业务侧结果就讲展厅**，两条我都能收成一句清楚的成就叙事。

# 现在回头看之前项目，还有哪些改进空间？

若 **单从数字展厅这类 AI 智能问答** 回头看，我会觉得还有几块值得做得更好：**第一，交互范式**：当时是轮询 + 占位较多，若换成 **流式输出（SSE/WebSocket）**，首字/首句的可感知延迟还能再压一截，空等心理会更好控。**第二，观测与实验**：埋点可以拆得更细——例如 **首 token 时间、各阶段放弃率、追问入口点击率、「看了依据仍不满意」的比例**，方便和算法一起做版本对比，而不是只看「接口成功率」。**第三，评测与回归**：可以沉淀一批 **固定问题集 + 期望 UI 状态**（加载中、部分输出、完整输出、失败兜底），做小范围的自动化或人工回归，避免模型一迭代界面行为「偶发翻车」。**第四，能力与规范**：追问、引用、免责声明、敏感场景兜底这些模式可以 **抽成可复用组件和规范**，下次换项目不必从零拼；若往后接入 **工具调用 / 多轮工具链**，前端还要提前想好 **调用过程怎么展示、出错怎么让人敢继续点**，这是我愿意接着补的一课。

# 总结一下你有什么优势（2-3 个），有什么不足？

**优势我会归纳成三条：**  
**第一，复杂前端交付 + 拿指标说话。** 长列表、Web 3D 这类高压场景我经历过完整闭环，习惯把「卡顿、崩溃、观感」落成 **可验收的数字**，而不是停在体感；出了问题也能 **拆链路、做回归**，适合 AI 产品里同样偏重的性能与稳定性。  

**第二，AI 问答类产品的前端经验 + 持续补 AI 工程。** 做过展厅场景下的等人、追问、可信展示，知道模型输出不确定时 **面上要怎么兜**；同时我也在系统补 Agent、评测这些新一代链路，能把 **用户体验** 和 **模型能力演进** 对齐着想。  

**第三，前端负责人 + SM 的双重角色。** 我能 **拆任务、盯外包与正编的交付质量**，也能 **站会、对齐阻塞、升级风险**——高压节点里更偏向 **把目标和责任说清楚**，减少团队内耗。  

**不足**：有时 **前期想把方案想得太完整**，容易占掉一部分迭代带宽。现在我刻意用 **时间盒 + 先最小可用再纵深** 约束自己；另外 **更大盘子（几十人）的组织管理** 我还在通过学习和旁观前辈案例补课，愿意在大厂里继续磨这一块。

# 别人对你的评价怎么样？最负面评价是什么？

正面评价一般是靠谱、抗压、推进力强，复杂问题不会往外甩——这和主人翁意识（非我莫属）比较接近。  
负面有时会听到「太较真」。我的复盘是：**较真要对准客户价值和关键路径**，而不是在无差别的细节上消耗协作信任；所以现在我会先锁优先级与时间盒，把「必须先赢的一次交付」说清楚，再深度打磨——既不糊弄客户，也不让同事难受。

# 举一个跨部门合作的案例

在数字展厅 AI 问答优化中，我和产品、算法、后端高频协作：产品要转化、算法要质量、后端要稳定——表面上 KPI 不同，但 **共同客户是终端用户**。我先把各方语言翻译成同一套可观测指标（等待时长、有效回答率、追问转化等），再落成统一会话、状态机和兜底策略；埋点数据 **双向透明** 给算法迭代，减少「我觉得」扯皮。这和「因为信任，所以简单」一致：**信息对齐了，协作成本就会下降**。最终体验和转化都有正向结果。

# 团队起冲突，怎么解决？（我们团队和其他团队：利益 / 优先级 / 资源）

这类一般是 **两边 KPI 不一致**：我们要的体验或交付节点，和对方要的稳定性、人力占用、排期不对付，表面上像在抢资源或甩锅。我的做法是：**先把各自的诉求写清楚**——我们要什么、什么时候要、对终端用户或收入的影响是什么；对方卡在哪（人力、风险、历史债务）。然后 **拉上能在优先级上拍板的人一起对齐**（通常是双方 TL 或产品经理），避免一线互相耗；会上尽量用 **同一套事实**（需求文档、埋点、故障影响面）说话，少用情绪词。若一时拍不死，我会争取 **拆里程碑**：有没有一块「交集」能先做掉、两边都能交差；或者 **小范围试点**，约定复盘时间再看要不要扩大——底线是 **不把交付悄悄押注在私下较劲上**，该升级就升级。

# 场景题：算法同学优先级更高，导致项目瓶颈，怎么解决？

这类瓶颈多半是 **「这条需求是不是全链路的最大优先级」** 没在主桌上说清楚，一线就容易互相觉得对方不配合。我会这么处理：

**先和业务 / 产品对齐一件事**：当前迭代里，**终端用户这条能不能等**、最晚要什么程度的可用——是全功能上线，还是先验证一波留存和转化。对齐完再讨论资源，而不是前端单方面催算法。

**再跟算法把依赖拆开**：哪些体验 **必须等模型迭代**（准确率、策略逻辑），哪些 **前端可以先兜一层**——比如交互闭环、会话状态、占位与流式反馈、规则或模板兜底、旧版本模型先顶着用、埋点把漏斗铺好。目标是 **面上先让用户「用得下去」**，算法版本到了再一键切换高质量能力，而不是整条链路干等。

**节奏上**：把「算法预计可人力的周」和「前端可交付的里程碑」 **写成依赖表**，固定同步；若冲突实在解不开，**升级给双方 Leader 拍优先级**，避免卡在私下较劲里。

我自己会 **少抱怨多同步**：进度和风险摊开讲，大家都对着同一套事实协作；前端该扛的交互与降级我会先补上，不把延期偷偷捂成团队矛盾。

# 你与团队起冲突，怎么解决？（我和团队内部同事：利益或分歧）

内部更容易卡在 **分工边界、方案取舍、绩效体感** 这些事上：你觉得你该主导，对方也觉得被抢了活；或者你认为质量红线不能退，对方认为进度更重要。我会 **先一对一聊清楚**：对方到底在在意什么（是被忽视、怕背锅，还是确实资源不够），有没有 **我表达太冲或信息不对称** 的地方；技术分歧就 **拿原型、数据、风险清单摊开**，欢迎对方挑刺，而不是在会上突然让人下不来台。如果真涉及 **利益分配说不拢**（比如署名、绩效归因），我不会拖着冷战，会 **请直属 Leader 介入做一次透明裁定**，大家按裁定执行，并把结论 **简短记下来**，避免下次重复撕同一件事。我自己这条线是 **对事不对人，决策下来了就一起把结果扛出去**。

# 我跟直属领导有摩擦怎么办？（常见：领导对我表现不太满意）

这种事我一般会 **先假定「人家不是在针对我，而是在看结果」**，提醒自己别一上来就辩解。接下来会做几步：**第一，把反馈听完整**，追问清楚——不满意到底是进度、质量、沟通方式，还是协作体感；最好落实到 **具体事例**，避免停在「你不行」这种空话。**第二，对齐预期**：你希望我这半个月 / 这个迭代 **肉眼可见地变成什么样**，我们能不能一起落成 **两三条可检查的行动项**（比如节点、指标、汇报节奏）。**第三，用行动回补信任**：按约定频率 **短同步**——不用等领导来盯，主动同步进展、风险和需要什么支持；交付上用 **可验证产出** 说话（如期上线的功能、数据变好、事故减少）。如果我觉得 **有误会**，我会挑一对一冷静摊开事实和数据，目的是 **对齐认知**，不是争输赢；如果确实是我 **能力或习惯上有缺口**，我会认账，该补课补课、该请人带我请人带。

**在什么情况下会考虑请 HRG 介入？** 一定 **不是第一步就绕过主管**——前提通常是：已经和直属主管做过 **多轮清晰的一对一**，标准仍然含糊，或者我感觉反馈 **缺少可追溯依据**、沟通容易卡在情绪里，靠自己双边打不开局面。这时我会 **主动约 HRG（HR 伙伴）**，把诉求说清楚：**我想对齐公司的规则和可行选项**，而不是来单纯吐槽老板。HRG 这边我能看到的是：**熟悉绩效与改进计划的流程**，能解释内部有哪些培训、辅导或协作机制可以接上；必要时也能以 **中性第三方** 协助我和主管把期望落实到 **书面框架**，减少对立。**我理解这正是 HRG 的价值**：既维护管理的一致性，也让员工 **被听见、被公正对待**——我愿意走这条正规渠道，而不是让小道消息发酵。

**底线**：长期拧巴且始终达不到清清楚楚的标准，再严肃考虑是否岗位不匹配——但在那之前，我会先把「听得懂反馈、改得动、跟得上」做到极致；需要组织介入时，我会优先信任 **HRG 的专业角色**。

# 最近有没有学新的技术？通过什么渠道学习？

最近重点在 AI 前端工程化、Web 性能和 3D 渲染优化。像秒悟这类产品我会亲自体验，从“用户输入到应用生成”全链路观察交互和工程实现。学习渠道主要是官方文档、技术博客、开源项目源码和高质量社区分享。

# 研究路线怎么确定的

我通常按“现象 -> 指标 -> 假设 -> 实验 -> 回归”来定路线。先确认瓶颈是 CPU、内存、网络还是主线程阻塞，再设计最小实验验证。验证有效才放大实施，实施后做回归和监控，确保不是局部优化、全局退化。路线不是一次定死，而是持续根据数据调整。

# 遇到了什么反尝试的问题，如何解决的

我做过一个反尝试：在 3D 场景里先尝试“统一降画质”来保性能，结果高端设备体验明显变差，业务方反馈很差。  
我及时复盘后改成“设备分级 + LOD + Worker 解码”的组合策略：低端优先保稳定，高端尽量保细节，中端平衡。这个反尝试让我认识到，性能优化不是单目标最优，而是分层策略下的整体最优。

# 工作中如何使用 AI？怎么看待 AI？

我平时的开发节奏，是按自己总结的 **Spec-Kit + Superpowers** 一套来的（详见仓库里的 `spec-kit-superpowers-开发方式总结.md`），用人话讲就是：**先把「做什么、为何做、怎么验收」写清楚，再让 AI 帮着高质量落地**。

**具体怎么用：**先用 **Spec-Kit** 这一路把东西钉死——项目底线（宪法）、功能规范（spec）、技术计划（plan）、再拆成 **可追踪的任务清单（tasks）**，避免「写到一半目标飘了」那种 vibe coding。任务下来了再交给 **Superpowers** 这一套执行习惯：**先摸清上下文再动手**，能 TDD 的就 **先测后写**，做完走 **审查 + 验证**，关键路径 **有证据再过合并关**。AI 在里面主要帮我扛 **样板代码、方案草稿、用例补充、文档同步** 这些重复劳动，但 **每条任务仍满足最小质量门槛**，不是图快凑合上线。

**怎么看待 AI：**我是 **积极拥抱** 的——工具和方法迭代快，我觉得是好事，愿意持续跟；本质上 AI 把 **个人产能和团队下限** 往上抬了一截，但 **方向盘和签字权还在人手里**：需求对不对、架构对不对、上线敢不敢背锅，仍然要人判断。对我来说，AI 不是替代工程师，而是 **逼着我们把规范、测试、验收做得更前置**——这和云上做 AI 产品要把体验做稳，是同一条思路。

落到工作上：**我对 AI 的态度是开放尝试 + 纪律兜底**；也愿意在阿里云智能这种节奏里，继续打磨这套 **「规范先行、验证收口」** 的协作方式。

# 之前工作强度如何？怎么看加班？

职业规划上会深耕 AI 前端，愿意扛核心业务与更高标准交付；我认为这和「今天最好的表现是明天最低的要求」是一致的：**你愿意在哪个水位线上重复自己**。  
工作强度上我经历过关键节点的冲刺，能接受阶段性加压，但我更看重 **目标清晰、避免无效卷**：复盘能减少重复劳动，就比单纯堆工时更符合长期产出。也认同「认真生活，快乐工作」——可持续的节奏才能在一次又一次技术浪潮里保持学习和交付质量；我会用计划和沟通减少低级消耗，而不是用熬夜感动自己。

# 您如何看待阿里当前的业务方向？

我粗浅的理解是，阿里现在抓 **云和 AI 绑在一起卖**，意思挺直白：**别让中小企业和开发者从零搭机房、啃一大堆配置**，而是把算力、存储、模型这些「重东西」收成他们 **伸手就能用的服务**，想法验证能快一点、上线能省事一点——这跟常说的「让天下没有难做的生意」我是能读出一条线的。我最近试了秒悟，感觉它不只是在「帮你写两段代码」，而是试着把 **模型、部署、存数据** 这些事串起来，让人 **说完需求更接近真能点开用的东西**。  
真要拼起来，大家比的肯定不只是模型有多大，还有 **交付稳不稳、用起来顺不顺、出问题好不好查**——说白了，用户等不等得起、信不信得过、敢不敢长期用。技术这块后面肯定还得变，前端正好是 **把新变化收成面上稳定体验** 的那一层，所以我个人对这个方向挺看好的。

我想进一步了解：秒悟当前阶段最核心的业务目标是拉新、留存还是商业化转化；前端与算法协作的机制和节奏；试用期内希望我优先解决的核心问题是什么。  
如果有机会加入，我很希望尽快进入关键链路，拿可量化结果来证明我的价值。

# 你能给公司带来什么价值？

我能带来的价值可以概括成三层，都围着「客户第一」落地。第一，把模型能力转成 **用户可感知、愿持续使用** 的体验：响应感、可信度、追问与兜底，而不是「接口通了就算交付」。第二，在复杂场景里 **扛得住高压、拆得清问题、拿得出指标**：长列表和 3D 的经历让我习惯分阶段交付、控制风险，关键时刻我能顶上把事情收口（此时此刻，非我莫属）。第三，通过规范和验证闭环 **降低团队协作摩擦**：边界写清楚、证据说了算，让迭代更简单（因为信任，所以简单）。  
一句话：**帮助团队把 AI 能力更快、更稳地变成客户侧结果**，并持续抬高体验基线。

# 如何向不懂技术的人解释云计算？

云计算可以理解成“把原本要自己买、自己维护的服务器和软件能力，变成像水电一样按需使用的服务”。  
你不用先投入很大成本搭机房，只要按业务需要开通资源，流量大就扩容，流量小就缩容，省钱也更灵活。

# 公司的 OKR 指标是什么？

我理解 OKR 一般会围绕业务增长、用户体验和交付效率三类目标。对前端岗位，我会主动把目标拆到可量化指标，比如页面性能、稳定性、关键转化漏斗和迭代周期。  
我希望做的是“把业务 OKR 映射成工程动作”，保证每个技术动作都能对应业务价值。

# 每个 O 完成的评价标准是什么？

我会用“结果 + 质量 + 可持续性”来衡量：结果是否达成核心指标，质量是否稳定可复用，可持续性上是否沉淀了机制和规范。  
例如优化项目不只看一次上线数据，还要看回归稳定性和后续团队复用效果。

# 反问环节：组织架构、试用期、转正标准、入职时间

我会这样反问：秒悟前端团队在应用生成链路里主要负责哪些核心模块（如生成态交互、发布链路、可观测）？试用期最核心的 1-2 个目标是什么，转正评估更看重哪些结果？  
如果双方确认匹配，我可以配合业务节奏尽快安排入职时间。我个人是非常希望加入这个团队的，也愿意在前期承担更高强度的攻坚任务。

# 技术更新快，你如何保持敏感度和先进性？

我默认「唯一不变的是变化」，所以会 **并行做输入与输出**：跟进官方与一线实践，同时每季度做 1～2 个可验证的小实验，而不是囤积教程。  
方法上用固定框架控风险：边界清楚 → 小步实现 → 测试与证据收尾；昨天有效的方法我会复盘是否仍是明天的下限（今天最好的表现是明天最低的要求）。只有跑出来指标变好，我才主张推广到团队，避免追新概念给客户和同事埋坑（客户第一）。

# 最近在关注哪些新技术？

我主要关注三块：AI 前端工程化（多轮会话、流式响应体验、Agent UI 形态）、性能与稳定性（Web Vitals、资源调度、监控告警）、3D/多媒体性能优化（渲染管线与跨端适配）。  
结合秒悟体验，我也在重点关注多模型协同产品中的前端问题，比如生成过程可视化、错误修复反馈、从“可生成”到“可维护”的体验设计。  
在 Agent 方向上，我正在持续学习和拆解 Agent Loop、Tool 调度、上下文工程、记忆机制、多 Agent 协作与评测体系，目标是把“能跑 Demo”提升到“可评估、可迭代、可上线”。
关注重点不是“新名词”，而是哪些技术能真实提升用户体验和业务结果。

# 入职后怎么快速适应公司节奏和业务？

我会用「30-60-90 天」：30 天吃透业务链路、关键指标和协作接口，多问少说猜；60 天交付 **一个可量化的小胜仗**（例如生成态等待体验或转化链路），证明我能创造价值；90 天沉淀可复用组件或规范，抬高团队下限（今天最好的表现是明天最低的要求）。  
全程主动对齐上级预期，优先级透明。我的风格是 **该扛的责任不往外推（此时此刻，非我莫属）**，但也尽量用文档和同步让协作简单。

# 你认为 AI 研发和传统开发的区别

传统开发更强调确定性流程和规则实现，AI 研发更强调概率性结果和持续迭代。  
能力要求上，AI 场景除了工程能力，还要求你有实验设计、效果评估和跨角色协作能力；工作内容上也更需要快速试错和数据闭环。  
我自己的理解是：传统开发偏“功能正确性”，AI 研发还要管理“上下文正确性”和“工具调用正确性”。所以我会额外关注 Context Engineering、Memory 设计和 Harness 验证，确保系统稳定输出而不是偶发可用。

# AI 的出现会让你焦虑吗？

不会回避焦虑，但我更把它当成升级信号。AI 会替代重复性执行，但也会放大“问题定义、系统设计、业务判断”的价值。  
我会持续学习并把 AI 融入工作流，让自己成为“会用 AI 提升交付质量和速度”的工程师，而不是被动等待变化。  
简历上 AI 标签不重，一部分是我 **有意取舍**：早期问答类产品的前端实现方式和当下爆款范式不完全一致，我不想用标题赢眼球、用细节却对不上预期。另一部分我会用 **持续投入** 补足：系统学习框架、沉淀开发体系、做小实验验证。把不确定性转成可证明的进步，是我一贯的应对方式。

# 简历上 AI 背景不重，你如何证明你能胜任 AI 前端岗位？

我会从三个层面来证明，落脚点都在 **客户可感知的结果**。第一是可迁移能力：高性能与复杂交互，本质是高压下的拆解、指标治理与稳定交付——AI 产品同样需要你扛不确定性（拥抱变化）。  
第二是真实实践：**公司里做过智能问答前端闭环**，简历写法不等于经历为零；我愿意在面试里说清楚业务贡献，并补足对流式、工具、Agent 等新链路的理解与实验（Spec-Kit / Superpowers 与各 Agent 模块的持续拆解）。  
第三是担当与结果：**入职后我会主动认领一条链路打到量化指标（此时此刻，非我莫属）**，用数据和复盘说话，而不是只强调「我会用工具」。这既对客户公平，也对团队简单透明。

# 试用期怎么度过？期望上级怎么带你？

**试用期我会当成「对齐预期 + 证明交付」的冲刺，而不是被动挨评。**  
**前几周**：尽快摸清 **业务链路、关键指标、协作接口和文档入口**，把「谁能拍板、找谁联调」画一张自己的心智图；主动约主管做一次 **目标对焦**，确认试用期里 **最重要的一两条产出**（例如某条体验链路、性能指标或交付节点），避免埋头做错优先级。**节奏上**：我习惯 **短同步**——按双周或团队惯例交付小结：做了什么、风险在哪、需要什么支持，不求表扬，先把 **透明度** 做起来。**产出上**：争取在试用期中段有一次 **可量化的小胜仗**（数据或业务能点头），后半段再沉淀文档/规范，证明 **可持续**。

**我期望上级怎么带：**  
**一是节奏**：固定 **1:1 或固定反馈窗口**，反馈尽量 **具体到行为与结果**，而不是泛泛「还不够好」。**二是边界**：优先级若有冲突，希望主管能帮我 **挡一层噪音**，让我专心啃试用期核心目标。**三是资源**：该见的协作方、该读的内部规范，主管 **点名引荐** 一次，比我自己摸索快很多。**四是里程碑**：若公司有 **书面试用期目标或转正口径**，希望和主管一起 **对齐成检查清单**，月底对照，减少「临门一脚才说不行」的体感。若涉及 **制度、福利、流程入口** 不清楚，我会 **按规矩找 HRG 对齐**，不把琐事都堆给业务主管。

# 绩效不好或被误解怎么办？

**第一步永远是「听清楚」**：绩效结果或反馈出来时，我先完整听完，追问 **具体事例、对照的标准是什么**——避免一上来就辩解，把对话变成情绪对抗。**第二步「对齐认知」**：若我觉得 **事实有出入**，我会一对一 **带好数据、交付记录、关键邮件** 说明，目的是 **校正误会**，不是推卸责任；若确实 **能力或习惯有短板**，我认账，和主管一起落到 **可执行的改进项**（例如哪类缺陷要清零、哪类协作要升级）。**第三步「书面留痕」**：重要共识我会习惯 **邮件或文档里简短 recap**，防止事后各说各话——这对双方都公平。**第四步「制度内升级」**：若多轮沟通仍 **标准含糊或长期无法对齐**，我会考虑 **请 HRG 介入**，借助正式流程把期望拉到 **可追溯的框架** 里，而不是私下抱怨或传闲话——我相信这也是 HRG **保障公平与规则** 的价值所在。

**心态上**：绩效不好可以是一次 **校正坐标的机会**；被误解则优先用 **事实与流程** 解开。底线是 **诚信交付 + 尊重规则**，既不躺平，也不搞对抗式维权。
