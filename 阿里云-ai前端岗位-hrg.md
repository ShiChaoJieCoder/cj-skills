# 研究路线怎么确定的
我通常按“现象 -> 指标 -> 假设 -> 实验 -> 回归”来定路线。先确认瓶颈是 CPU、内存、网络还是主线程阻塞，再设计最小实验验证。验证有效才放大实施，实施后做回归和监控，确保不是局部优化、全局退化。路线不是一次定死，而是持续根据数据调整。

# 遇到了什么困难，如何克服的
我遇到过的最大困难之一是 H5 长列表在低端机严重卡顿：一次性渲染导致 DOM 过多、图片解码挤占主线程、滚动掉帧明显。仅做虚拟列表仍会在快速滑动时被大图并发加载拖慢。  
我把问题拆成三线治理：虚拟列表控制 DOM 数量，`IntersectionObserver` + `rootMargin` 做预加载，`decoding="async"` 和定高骨架控制解码阻塞与 CLS。最后把性能从“可用边缘”拉到“稳定可用”，FPS 提升到 55+ 量级，内存明显下降。

# 遇到了什么反尝试的问题，如何解决的
我做过一个反尝试：在 3D 场景里先尝试“统一降画质”来保性能，结果高端设备体验明显变差，业务方反馈很差。  
我及时复盘后改成“设备分级 + LOD + Worker 解码”的组合策略：低端优先保稳定，高端尽量保细节，中端平衡。这个反尝试让我认识到，性能优化不是单目标最优，而是分层策略下的整体最优。

# 所负责的项目中哪个最有挑战或有成就感？（从业务角度具体展开）
最有成就感的是数字展厅 AI 问答体验优化。初期核心问题是用户“等不到反馈”、答案可信感弱、追问链路断。  
我做了三层优化：感知层用“先反馈后回填”的状态机降低空等焦虑；质量层通过 `chatSessionId` 复用上下文并展示参考资料提升可解释性；业务层通过追问机制和边界控制拉长会话链路。结果是用户等待焦虑下降、有效回答感知更稳定、连续追问和有效咨询转化更好。

结合秒悟体验，我也在重点关注多模型协同产品中的前端问题，比如生成过程可视化、错误修复反馈、从“可生成”到“可维护”的体验设计。  
在 Agent 方向上，我正在持续学习和拆解 Agent Loop、Tool 调度、上下文工程、记忆机制、多 Agent 协作与评测体系，目标是把“能跑 Demo”提升到“可评估、可迭代、可上线”。
关注重点不是“新名词”，而是哪些技术能真实提升用户体验和业务结果。

# 入职后怎么快速适应公司节奏和业务？
我会用“30-60-90 天”节奏：前 30 天快速熟悉秒悟业务链路和关键指标、补齐关键人脉；60 天交付一个可见优化点（例如生成态等待体验或发布转化链路）；90 天形成可复用的方法或组件。  
同时会主动对齐上级预期，确保工作优先级和团队目标一致。  
我的目标不是“适应就好”，而是尽快在核心场景打出结果，成为团队可以放心托付关键模块的人。

# 场景题：算法同学优先级更高，导致项目瓶颈，怎么解决？
我会先和产品一起明确该需求的业务优先级，争取统一排序；再和算法同学拆分“必须算法支持”和“前端可先落地”的部分并行推进。  
例如先做前端交互闭环、会话管理、兜底策略和埋点，先跑通最小可用版本，等算法资源到位再迭代质量。这样既尊重资源现实，也不让项目停摆。

# 你认为 AI 研发和传统开发的区别
传统开发更强调确定性流程和规则实现，AI 研发更强调概率性结果和持续迭代。  
能力要求上，AI 场景除了工程能力，还要求你有实验设计、效果评估和跨角色协作能力；工作内容上也更需要快速试错和数据闭环。  
我自己的理解是：传统开发偏“功能正确性”，AI 研发还要管理“上下文正确性”和“工具调用正确性”。所以我会额外关注 Context Engineering、Memory 设计和 Harness 验证，确保系统稳定输出而不是偶发可用。

# AI 的出现会让你焦虑吗？
不会回避焦虑，但我更把它当成升级信号。AI 会替代重复性执行，但也会放大“问题定义、系统设计、业务判断”的价值。  
我会持续学习并把 AI 融入工作流，让自己成为“会用 AI 提升交付质量和速度”的工程师，而不是被动等待变化。  
坦白说，简历上没有特别重的 AI 标签确实会带来压力，但我应对焦虑的方式是持续投入：系统学习框架、做方法论沉淀、用小项目验证。把焦虑转成行动，是我处理不确定性的核心方式。

# 简历上 AI 背景不重，你如何证明你能胜任 AI 前端岗位？
我会从三个层面来证明。第一是可迁移能力：我过去做的高性能前端和复杂交互优化，本质上就是在高不确定场景下做问题拆解、指标治理和稳定交付，这些能力在 AI 产品里同样关键。  
第二是真实实践：我不仅做过 AI 问答体验优化，也长期在自学和实践 AI 工程方法，像 Spec-Kit + Superpowers、Agent Loop、Tool System、Context Engineering、Memory、Multi-Agent、Harness Engineering，这些我都在持续做拆解和小实验，不是停留在概念层。  
第三是结果导向：我不会把“会用 AI 工具”当终点，而是强调“能把 AI 能力转成稳定可用的产品体验和业务指标”。如果加入团队，我会优先拿一个关键链路做出可量化优化结果，用结果来证明匹配度。

# 遇到过的最大的困难和挫折，如何克服的？（体现“皮实”）
我遇到过最大的挫折，是一个 3D 场景项目临上线前一周问题集中爆发：低端安卓频繁崩溃，高端机又反馈画质下降，业务发布节奏已经排定，时间和质量压力同时压上来。那段时间白天对齐方案、晚上压测复盘，连续多天到凌晨，团队情绪也很紧张。最困难的是我们前期还走过弯路，比如尝试过一刀切降画质，崩溃少了但高端体验明显变差，等于只是换了一个问题。  
后来我没有被情绪带着走，而是把问题拆成两条线：“能不能打开”和“能不能流畅”。先建立设备分级策略，再引入 LOD 按距离切换模型精度，同时把重材质解码迁到 Worker，减少主线程阻塞。我们按日做小步回归，把关键指标持续拉回，最终按节点完成上线。  
这件事让我真正成长的是：我不是顺风局才有表现，越复杂、越高压的时候，我越能把问题拆清楚、把节奏稳住、把团队往前推。我理解的“皮实”不是硬熬，而是在很难的情况下依然保持乐观和执行力，最后把结果做出来。
