# AI 辅助前端开发：面试高分表达稿（React + Spec-Kit + Superpowers）

> 用途：这不是“技术文档”，而是一份**面试可直接复述**的表达手册。  
> 目标：把“我会用 AI”升级为“我能把 AI 纳入工程体系，稳定交付”。

---

## 1. 先给面试官的总框架（15 秒）

我把 AI 协作拆成两层：

- **上层：Spec-Kit（管方向）**  
  先定义做什么、为什么做、做到什么算完成，防止需求漂移。
- **下层：Superpowers（管质量）**  
  实现时强制测试、审查、验证闭环，防止代码质量失控。

一句话就是：**Spec-Kit 定边界，Superpowers 保质量。**

---

## 2. 1 分钟口播版（面试最常用）

我在 React 项目里用 AI，核心不是“写更快”，而是“写得准、写得稳”。  
方法上我会先做规范层：用 Spec-Kit 写 `constitution/spec/plan/tasks`，把需求范围、验收标准、技术边界都固定下来，避免 AI 中途改路由、乱加状态、引入不必要依赖。  
然后进入执行层：用 Superpowers 按任务推进，每个任务都走“先写测试、最小实现、跑通验证、代码审查”的闭环。  
我自己的经验是，这种方式在多人协作里很有用：沟通成本更低，需求可追踪，代码质量也更稳定。对面试来说，我想表达的是——我不是把 AI 当自动生成器，而是把它当有护栏的工程加速器。

---

## 3. 3 分钟结构化回答（问“展开讲讲”时用）

### 3.1 我解决的核心问题

AI 在前端开发里最容易出现三个问题：

- **需求漂移**：写到一半改目标，最终交付和业务不一致。
- **结构失控**：状态乱放、目录混乱、样式污染，后期难维护。
- **质量缺口**：能跑但不稳，缺测试、缺审查、缺验证。

所以我不会直接“让 AI 写代码”，而是先建立流程护栏。

### 3.2 我的方法论（两层治理）

- **规范层（Spec-Kit）**：管“做什么”
  - `constitution`：全局工程原则（质量、安全、性能、测试底线）
  - `specify`：业务目标和验收标准（What/Why）
  - `plan`：技术方案和架构边界（How）
  - `tasks`：可执行任务分解
- **执行层（Superpowers）**：管“怎么做才靠谱”
  - 任务级 TDD（红-绿-重构）
  - 子代理并行（复杂需求提速）
  - 代码审查与完成前验证（上线前质量闸门）

### 3.3 我在前端里的落地方式（React 项目）

- 页面和路由先定清楚，再让 AI 逐页交付（纵向切片）
- 状态边界先定义（局部状态 vs Zustand 全局状态）
- 请求层统一放 `services`，避免组件里到处写裸请求
- 样式采用 SCSS Modules 或明确命名约定，防全局污染
- 每轮改动可运行、可验证、可回滚，不做“超大一次性提交”

### 3.4 我怎么衡量它真的有效

- 返工次数下降（需求变更引发的大改减少）
- 回归 bug 下降（测试与审查前置）
- 交付节奏更稳（每个切片都可演示、可验收）

---

## 5. 把“9 步”重写成面试可讲的“三段九步”

> 面试可直接说：**先定图纸（1-3），再打地基（4-7），最后立规矩（8-9）。**

### 阶段 A：定图纸（1-3）

1. **明确 MVP**：用户是谁、核心价值是什么、第一版必须交付什么。  
2. **写验收标准**：每个关键路径都可验证，不写模糊目标。  
3. **定页面与动线**：路由表、页面职责、核心交互先锁定。

### 阶段 B：打地基（4-7）

4. **明确边界**：这期不做什么，防 AI“加戏”。  
5. **锁技术栈**：React + Zustand + SCSS + Vite（或 Webpack）保持一致。  
6. **定目录与分层**：`pages/components/features/stores/services` 职责清晰。  
7. **定数据与状态策略**：哪些进 store，哪些留在页面局部状态。

### 阶段 C：立规矩（8-9）

8. **定编码与提交流程**：小步提交、每步可运行、禁止一次改太多。  
9. **定质量闸门**：测试通过、代码审查通过、验证通过再进入合并。

---

## 6. 你和面试官的“映射关系”讲法（加分点）

把三段九步映射到 Spec-Kit + Superpowers：

- **步骤 1-3** 对应 Spec-Kit 的 `constitution/specify`
- **步骤 4-7** 对应 Spec-Kit 的 `plan/tasks`
- **步骤 8-9** 对应 Superpowers 的 TDD + review + verification

这句话可以直接说：

> 我的 9 步不是零散 checklist，而是和 Spec-Kit + Superpowers 一一对应的工程流水线。

---

## 7. 面试官高频追问：标准回答

### Q1：为什么要这么重？小项目也需要吗？

小项目用轻量版：`spec + tasks + 基础测试` 就够。  
复杂项目才走完整链路。核心不是“重”，而是“防返工”。

### Q2：AI 已经很强了，为什么还要规范？

AI 强在生成速度，不等于理解业务边界。  
规范的价值是把“意图”变成“可执行约束”，减少偏航成本。

### Q3：你怎么保证不是形式主义？

我看结果指标：返工率、回归缺陷、交付周期。  
如果指标没改善，我会压缩流程，不会为流程而流程。

### Q4：你在团队协作里怎么推行？

先从一个模块试点，保留最小闭环：任务分解 + 测试 + 审查。  
跑通后再推广到更多模块，不一刀切。

---

## 8. React 栈“护栏清单”（面试可顺口说）

- 函数组件 + Hooks，避免混用旧模式  
- 全局状态用 Zustand，按业务域拆 store  
- API 统一走 `services`，不在组件里散落请求  
- 样式优先 Modules，减少全局选择器污染  
- 环境变量遵循前端暴露规则，不把敏感信息打进包  
- 提交前至少过 lint、核心路径测试、构建验证

---

## 9. 一个可直接讲的实战案例模板（优惠券功能）

你可以这样描述：

1. 我先用 Spec-Kit 定义优惠券功能的范围、规则和验收标准；  
2. 在 `plan` 里明确数据模型、接口契约和异常处理；  
3. 用 `tasks` 拆成“模型/服务/API/页面交互/测试”几个子任务；  
4. 实现阶段用 Superpowers 逐任务执行 TDD 和代码审查；  
5. 最后按质量闸门验收并合并。  

结果是：需求和实现一致性更高，后续扩展（比如叠加规则、活动策略）也更容易。

- **我用 AI 的重点不是“多快写完”，而是“在可控边界内稳定交付”。**
- **Spec-Kit 让我不跑偏，Superpowers 让我不掉质。**
- **流程不是束缚效率，而是把返工前置消灭。**

# Vibe Coding 项目开发模板（前端 · React 栈）

> **方法论来源**：清华姜学长 VibeCoding 思路  
> **默认技术栈**：React（Hooks）+ Zustand + SCSS + Vite 或 Webpack  
> **用途**：用 AI 写前端时的一套「先想清楚、再小块交付、最后用人把关」的流程；适合写进简历、**面试时讲清楚你怎么协作 AI**。

---

## 先说清楚：Vibe Coding 是什么（通俗版）

很多人用 AI 写代码，是「想到一句说一句」，结果容易返工：界面改来改去、全局状态乱长、样式互相覆盖。**Vibe Coding** 强调的是：**先把需求、页面、技术边界写成「看得见、对得上」的约定，再让 AI 在约定里一小块一小块地写**；人负责验收、规范和安全，AI 负责提速。

你可以跟面试官这样概括：**不是让 AI 代替思考，而是用流程把 AI 的输出「管住」**。

---

## 给面试官怎么讲（结构建议）

| 环节 | 你可以说什么 |
| :--- | :--- |
| **开场** | 我在个人/项目里用「需求先行 + 小步验收 + 规范兜底」的方式做 AI 辅助开发，技术栈是 React + Zustand + SCSS + Vite/Webpack。 |
| **痛点** | 直接说：最怕 AI 一次改太多文件、乱加依赖、或者把敏感信息写进前端。所以我会先定 PRD、路由和状态边界，再切片开发。 |
| **做法** | 分三步讲：**定图纸**（需求、PRD、页面框架）→ **打地基**（边界、技术栈、目录、文档路径）→ **立规矩**（代码规范、Git、质量检查）。 |
| **亮点** | 强调三件事：**需求与验收写清楚**（可对齐测试）、**按页面/路由纵向切片**（每步都能点开页面验收）、**Hooks / Zustand / 环境变量有硬性约定**（可维护、安全）。 |
| **收尾** | AI 生成的代码我会过 ESLint、看关键路径、确认没有密钥进仓库；复杂问题先收集报错再让 AI 改，避免空转。 |

---

## 面试 30 秒版 & 1 分钟版（口语化）

**30 秒（背这一段就够用）**：

> 我用 AI 写前端时，会先定 PRD、路由和 Zustand 的边界，再按**页面切片**让 AI 改，每一块都能在浏览器里验收。样式用 SCSS Modules、全局状态按领域拆 store，配合 ESLint 和构建配置，避免 Hooks 乱用、状态乱堆、环境变量把敏感信息暴露出去。

**1 分钟（面试官让「展开说说」时用）**：

> 我不把 AI 当「一次性生成整站」的工具，而是当**在明确边界里的加速器**。开工前我会把需求、验收标准、路由表和哪些数据进全局 store 写清楚；开发时一次只做一条能走完的用户路径，比如登录再首页，避免一大坨代码不好测。代码层面我约定 React 只用函数组件和 Hooks，请求集中在 services，Zustand 按业务拆文件。最后上线前会跑 lint、build，敏感信息只放服务端，前端配置里不写真正的密钥。这样**返工少、协作也好讲清楚**。

## 名词秒懂（面试官听到英文时可以顺口解释）

| 词 | 白话 |
| :--- | :--- |
| **Spec-first** | 先写清「做什么、怎样算做完」，再写代码。 |
| **纵向切片** | 按一条完整用户路径（例如从进到某个页面到点按钮）切任务，而不是只先写一堆底层工具函数。 |
| **Guardrails（护栏）** | 用规范、目录、ESLint、环境变量约定限制 AI，防止它乱改风格、乱加库、泄露密钥。 |
| **MVP** | 最小可上线版本：必须先有的功能；其他的排二期。 |

## 本栈专属：写在 Rules / 对话里的「硬约束」（可复制）

下面整段可以放进 **Cursor Rules**，或者每次开任务复制给 AI。**面试可以说**：这是我和 AI 的「合同」，减少它随意换技术方案。

```text
- 框架：React 18+，只用函数组件 + Hooks；类组件除非维护旧代码否则禁止新建。
- 状态：全局用 Zustand；局部 UI 状态优先 useState/useReducer；不要用 Redux 除非项目已有。
- 样式：SCSS；优先 CSS Modules（*.module.scss）或约定 BEM；禁止随意全局污染（慎用无命名空间的裸全局选择器）。
- 构建：以 Vite 或 Webpack 之一为准；环境变量遵循项目约定（Vite 用 import.meta.env + VITE_ 前缀）。
- 请求：API 封装在统一 services/api 层；组件内不写裸 fetch 到处复制。
```

**一句话解释**：前端**看得见打包结果**，所以密钥、数据库密码绝不能写进会被下载的 JS；Vite 里只有带 `VITE_` 的变量会进浏览器，这一点要在团队里统一口径。

### 阶段 2：打地基（步骤 4–7）

> **目标**：构建工具、目录、数据从哪来、状态放哪，一次性讲明白，避免同一个项目里混两种写法。

#### 步骤 4：项目边界 & 非功能需求

- **明确不做**（防止 AI 多加页面、多加库）：
- **性能**：要不要路由懒加载（`React.lazy`）、长列表要不要虚拟列表（可以写「本期不做，二期做」）：
- **兼容**：最低浏览器版本；一般不再支持 IE：
- **部署**：是否 CDN、子路径部署（base）、用 `history` 还是 `hash` 路由：

**Vite 注意**：

- 只有 `VITE_` 开头的变量会进前端；不要用错。
- 本地联调后端常用 `vite.config` 里的 `server.proxy`。

**Webpack 注意**：

- 通过 `DefinePlugin` 等写进前端的值都会进包，**不能写真正的密钥**。
- `publicPath` 要和线上资源路径一致。

**面试一句话**：**边界写清楚**，AI 才不容易「帮你加戏」；性能、兼容、部署这几条是前端面试常问的，你提前想清楚会显得稳。

**Superpowers**：**`superpowers:writing-plans`** — 在计划中写清 **不做功能、NFR、部署/base path/环境变量暴露规则**；与 Skill 内 Scope 自检与「无 TBD」一致。调用：`/` → `writing-plans`。

#### 步骤 5：锁定技术栈（前端默认已填，可改）

| 分类 | 技术选型 | 备注 |
| :--- | :--- | :--- |
| 框架 | React（Hooks） | 函数组件；副作用写在 `useEffect` 里并写全依赖 |
| 路由 | React Router（v6+）或项目约定 | 路由集中配置，改起来不容易乱 |
| 全局状态 | Zustand | 按业务拆成多个 store 文件，避免一个文件无限变大 |
| 样式 | SCSS + CSS Modules（推荐）或 BEM | 颜色间距尽量用变量 |
| 构建 | **Vite** 或 **Webpack**（二选一） | 全项目统一一套，不要混 |
| HTTP | axios / fetch 封装 | 统一处理错误码、Token（若有） |
| 类型（可选） | TypeScript | 若采用，在规则里写清新文件用 `.ts` / `.tsx` |
| 测试（可选） | Vitest + RTL / Jest | Vite 项目常用 Vitest |

**Zustand 边界示例（写给 AI 看的「一句话」）**：

- 例：「登录态放在 `useUserStore`；列表筛选可以放在页面 state 或 `useListStore`；**不要把整份服务端分页列表长期堆在全局**除非多个页面要共享。」

**面试一句话**：选型要能说出 **一句为什么**（例如：Zustand 轻量、和 Hooks 配合简单；Vite 开发体验好），比单纯列名词分更高。

**Superpowers**：**`superpowers:test-driven-development`** — 在锁定依赖时同时固定 **测试命令**（如 `npm test` / `vitest`）；插件强调红-绿-重构，**不强迫**每个业务都 TDD，但 CI/提交前应有可跑测试与失败可见。调用：`/` → `test-driven-development`。

#### 步骤 6：AI 轻量架构草案（前端向）

**推荐目录示例（可按项目裁剪）**：

```text
src/
  app/                 # 入口、providers、路由出口
  pages/               # 页面级（与路由一一对应）
  components/          # 通用 UI（无业务或弱业务）
```

### 阶段 3：立规矩（步骤 8–9）

> **目标**：让 AI **照着你们项目里已经有的好例子**写，而不是每次换一套状态管理或目录风格。

#### 步骤 8：开发规范 + 参考资料夹

**React / Hooks 规范（示例）**：

- 组件文件：`PascalCase.tsx`；自定义 hook：`useXxx.ts`。
- `useEffect`：依赖写全；有订阅、定时器要清理。
- 避免在渲染里不断创建新对象/新函数，导致子组件无意义重渲染（需要时再用 `useMemo` / `useCallback`，别过度优化）。
- 列表要有稳定的 `key`；条件渲染注意 `0`、`''` 这种容易踩坑的值。

**Zustand 规范**：

- 一个业务领域一个 store 文件，例如 `stores/userStore.ts`。
- 复杂能力用中间件（如 devtools、persist）时，在文件顶部统一配置。
- 组件里尽量 **按字段订阅**（`useXxxStore(s => s.token)`），避免整 store 一订阅，无关更新也触发渲染。

**SCSS 规范**：

- 组件样式：`ComponentName.module.scss`，class 命名团队内统一（camelCase 或 BEM 选一种）。
- 全局样式只放 reset、变量等；不要在单个组件里一股脑 `@import` 巨大全局样式。
- 颜色、间距用变量，少写「魔法数字」。

## 三、开发过程 5 个核心关键点（前端加强版）

### 1. 小步迭代 + 端到端切片

> 每一小块都要满足：**能打开对应路由、主流程能点通、样式别整页崩**，再开始做下一块。

- 切片 1：  
  - 验收：✅ / ❌  
- 切片 2：  
  - 验收：✅ / ❌  

**建议顺序**：先把页面架子搭出来 → 用假数据把 UI 跑通 → 再接真实接口 → 再补加载失败等状态 → 最后再抠性能和边界情况。

**面试一句话**：这和敏捷里说的 **纵向切片**很像：**先交付能演示的一整条路径**，而不是底层写一堆却迟迟联调不上。

**Superpowers**：**`superpowers:executing-plans`**（或 **`superpowers:subagent-driven-development`**，任务独立且环境支持子代理时）— 每个切片对应计划中的一组任务，按检查点执行与验证。调用：`/` → `executing-plans` 或 `subagent-driven-development`。

---

### 2. 强制拆分模块，拒绝「一坨代码」

> **页面负责拼装**；复杂逻辑放到 **hooks / services**；跨页面共享的数据进 **stores**。

- 页面：`src/pages/...`  
- 展示组件：`src/components/...`  
- Store：`src/stores/xxxStore.ts`  
- 样式：`xxx.module.scss`  

**面试一句话**：文件职责清楚，**Code Review 和以后改需求**都更轻松；也方便向面试官 walk through 你的项目结构。

**Superpowers**：**`superpowers:writing-plans`** — 在计划中 **锁定文件职责与边界**（`writing-plans` 的 File Structure / 单文件单责），从根上避免「一坨代码」。调用：`/` → `writing-plans`。

---

### 3. 限制 AI 擅自修改：状态摘要（前端模板）

每次开新一轮对话前，可以把下面这段贴在提示最上面，告诉 AI **现在做到哪、这次只许动哪**：

```text
构建工具：Vite（或 Webpack），Node 版本：x.x
当前完成：页面/路由 xxx，store：xxx
本轮任务：xxx
禁止修改：webpack/vite 配置（除非明确说改）、已有 store 的对外 API
样式约束：只用 CSS Modules，类名前缀 xxx
```

**面试一句话**：这叫 **上下文管理**——AI 不知道你上一条聊了什么，**你用几句话把边界重复一遍**，比让它自由发挥更稳。

**Superpowers**：**`superpowers:using-superpowers`** — 每轮对话前先检索是否有适用技能；把本模板贴在用户消息顶部作为 **显式约束**（该 Skill 要求：用户意图优先，但流程技能优先于默认行为）。调用：`/` → `using-superpowers`。

---

### 4. 死守安全底线：密钥不进前端

- **服务端密钥、数据库密码**不要出现在 React 源码里，也不要被打进用户能下载的 JS 包里。
- **Vite**：只有 `VITE_` 开头的会暴露到浏览器；真正敏感的配置放后端。
- **Webpack**：写进 `DefinePlugin` 的值同样会进客户端，**不能当保险箱用**。
- **用户 Token**：按团队规范放在内存、httpOnly Cookie 或短期 + 刷新；不要轻易默认「长期塞 localStorage」，除非你们评估过风险。

**面试一句话**：前端代码任何人都能下载，**默认当公开**；敏感能力走后端或网关，这是基础安全观。

**Superpowers**：**`superpowers:verification-before-completion`** — 对「无密钥进 bundle / build 无泄露」类结论，须用 **可重复命令**（如 `rg`/`npm run build`/审计脚本）**当场跑一遍再下结论**。调用：`/` → `verification-before-completion`。

---

### 5. 科学应对报错：两次无新证据就停

> 报错时先把 **完整报错信息、哪个文件哪一行、浏览器和 Node 版本** 收集齐，再让 AI 改；如果两轮都没有新信息，先换思路（查文档、最小复现），不要原地重复问。

- 构建失败：分清是 **ESLint**、**TypeScript**、还是 **SCSS / 静态资源路径** 问题。
- 运行时：注意 React 18 **严格模式**下 effect 会执行两次、以及 **闭包拿到旧 state** 这类经典问题，有时要先自查再甩给 AI。

**面试一句话**：体现的是 **调试习惯**：用证据驱动，而不是和 AI 互相猜。

**Superpowers**：**`superpowers:systematic-debugging`** — 按结构化步骤缩小问题，用 **证据**（完整栈、复现、命令输出）驱动，避免无新信息的反复尝试。调用：`/` → `systematic-debugging`。

## 四、项目验收与收尾（前端清单）

- [ ] 功能和 PRD、路由表一致
