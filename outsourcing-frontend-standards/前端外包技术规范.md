# 前端外包技术规范（React Hooks + SCSS + Vite + Zustand）

## 1. 目标与适用范围

本规范用于约束外包团队前端交付，统一技术选型、开发方式与安全基线，降低项目维护成本与安全风险。  
适用于所有新建页面、组件、业务模块和公共能力改造。

## 2. 强制技术栈（必须满足）

- 框架：`React`（函数组件 + Hooks）
- 构建工具：`Vite`
- 样式方案：`SCSS`（禁止使用 Less/Stylus 等其他预处理器）
- 状态管理：`Zustand`
- 语言：`TypeScript`（建议严格模式，减少隐式 any）

## 3. 开发规范（核心要求）

### 3.1 React Hooks 规范

- 仅允许函数组件，禁止新增 class 组件。
- 组件内副作用统一使用 `useEffect` / `useLayoutEffect` 管理。
- 自定义 Hooks 统一放在 `hooks` 目录并以 `use` 开头命名。
- Hooks 依赖项必须完整，不允许故意绕过依赖检查。

### 3.2 SCSS 规范

- 样式文件统一采用 `.module.scss` 或明确约定的分层目录。
- 禁止内联大段样式，减少样式与逻辑耦合。
- 颜色、间距、字号等尽量抽取为变量或 design tokens。

### 3.3 Vite 工程规范

- 使用 Vite 默认 ESM 工作流，禁止引入与 Vite 冲突的旧式打包链路。
- 按页面或模块进行懒加载，避免首屏包体积过大。
- 环境变量必须通过 `import.meta.env` 访问，禁止硬编码敏感地址和密钥。

### 3.4 Zustand 状态规范

- 全局状态仅存跨页面/跨模块共享数据。
- store 按业务域拆分，避免单一超大 store。
- 禁止将临时 UI 状态全部堆入全局状态（优先局部 state）。

## 4. 安全基线：禁止的高风险 JavaScript 写法

以下写法存在明显安全隐患，必须禁止：

### 4.1 代码执行与动态注入类

- 禁止使用 `eval()`
- 禁止使用 `new Function()`
- 禁止字符串参数形式的 `setTimeout("...")` / `setInterval("...")`
- 禁止使用 `document.write()`

### 4.2 DOM 注入与 XSS 风险类

- 禁止直接拼接不可信内容到 `innerHTML`
- 禁止在 React 中使用 `dangerouslySetInnerHTML` 渲染未经严格净化的数据
- 禁止拼接脚本片段后动态插入 DOM 执行

### 4.3 原型污染与对象安全类

- 禁止直接信任并合并外部输入对象到业务对象（需白名单字段）
- 禁止使用可能污染原型链的危险合并方式（如不受控深拷贝）
- 禁止在业务代码中修改内置对象原型（`Object.prototype` 等）

### 4.4 URL 与跳转类

- 禁止直接拼接不可信 URL 并执行跳转
- 禁止对不可信链接使用 `window.open` 且不加安全参数
- 外链打开必须包含 `noopener,noreferrer`（或等效安全策略）

### 4.5 敏感信息暴露类

- 禁止在前端硬编码密钥、令牌、证书、私有接口地址
- 禁止将敏感信息打印到控制台（生产环境）
- 禁止在本地存储中明文持久化高敏感数据（如长期 token）

## 5. 推荐的工程化强制手段

为确保规范可执行，建议在外包仓库默认开启如下约束：

- `ESLint`：开启 `no-eval`、`no-implied-eval`、`no-new-func` 等规则
- `typescript-eslint`：限制不安全类型与不安全调用
- `pre-commit`：提交前自动执行 lint 与基础安全扫描
- `CI`：PR 必须通过 lint/test/build/security check 才可合并

## 6. 外包交付验收清单（建议）

- 技术栈符合：React Hooks + SCSS + Vite + Zustand
- 不存在本规范第 4 节禁用写法
- lint 无 error，安全扫描无高危项
- 核心流程具备基本测试与异常兜底
- 代码结构可维护（模块边界清晰、命名统一、注释必要且简洁）

## 8. 如何使用本外包 Skill（落地指引）

本目录已提供可直接复用的 skill 文件：`SKILL.md`。  
在 Cursor/Agent 场景下，可将其作为“外包交付审查器”使用。

### 8.1 使用时机（建议触发点）

- 外包提交新页面、新模块、重构方案后
- 外包提测前（预验收）
- 代码评审（Code Review）阶段
- 准备上线前做一次安全与规范兜底检查

### 8.2 快速调用方式（可直接复制）

你可以在对话中直接下达类似指令：

- `请按 outsourcing-frontend-standards/SKILL.md 审查这次外包提交，给出不合规项。`
- `用外包 skill 检查本次改动是否满足 React Hooks + SCSS + Vite + Zustand。`
- `按外包 skill 扫描是否有安全红线写法，并输出整改清单。`

### 8.3 标准评审输出模板（建议统一）

建议每次评审输出包含以下字段，便于沉淀与追踪：

- 评审结论：通过 / 有条件通过 / 驳回
- 技术栈符合度：是否满足 React Hooks + SCSS + Vite + Zustand
- 安全红线检查：是否命中禁用写法（逐条列出）
- 工程检查：lint / test / build / security check 结果
- 整改项：问题描述 + 严重级别 + 修改建议 + 截止时间

### 8.4 推荐执行节奏（管理视角）

1. **需求评审后**：先发外包“技术栈与禁用清单”，防止方向跑偏。  
2. **开发中期**：做一次抽样检查，提前发现不合规写法。  
3. **提测前**：按 skill 做完整检查并形成整改单。  
4. **上线前**：仅对高风险项复检，确认无红线问题残留。  

### 8.5 与本规范文档的关系

- `SKILL.md`：用于“快速执行检查”（短平快、可重复）
- `前端外包技术规范.md`（本文）：用于“完整条款说明”（对齐标准、培训、归档）

建议策略：**平时用 Skill 快速审查，争议项回到本文逐条判定。**
