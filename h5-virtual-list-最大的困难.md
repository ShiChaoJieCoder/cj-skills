| 维度 | 内容 |
|------|------|
| **情境（S）** | 列表 **500+ 条**，每条有高清图、文案；**一次性渲染 DOM 上万**、滚动时节点还堆积，**TTI 超 2s、内存 200MB+、FPS 掉到 20 左右**，低端机 **白屏感、卡死**；未进视口的图也在 **浪费流量**。 |
| **任务（T）** | 把 **DOM 数量、内存、解码与滚动** 拆开治理，让滑动在移动端 **稳定 55+ FPS** 量级。 |
| **困难点（核心矛盾）** | 只做虚拟列表不够，**快速滑动时大图并发 + 主线程图片解码** 仍会 **顿卡**；骨架与真实内容高度不一致又会 **CLS**。 |
| **行动（A）** | **虚拟列表**：幽灵占位撑滚动条、`startIndex/endIndex` 只渲染可视区 + **buffer**，列表用 **`transform: translateY`** 做偏移，滚动用 **`requestAnimationFrame`** 对齐刷新。**图片**：`IntersectionObserver` + `rootMargin` 预加载；`decoding="async"`；LQIP / `aspect-ratio` 控 **CLS**。**组件**：如 `React.memo` 减少滚动时无效渲染。 |
| **结果（R）** | 文档口径：**DOM 从 10,000+ 压到约 50～100 内**；**FPS 约 55+**；**内存从 200MB+ 降到约 30MB**（数字请按你实际上线再微调）。 |

**一句话收口：** 「最难的是 **长列表同时吃 DOM、带宽、解码三条线**：虚拟列表 **减 DOM**，懒加载 + 异步解码 **减主线程压力**，定高骨架 + 宽高比 **控 CLS**，几轮迭代才把 FPS 拉稳。」
