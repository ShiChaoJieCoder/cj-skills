| 维度 | 内容 |
|------|------|
| **情境（S）** | Web 端 **机型差异极大**：低端安卓 **显存溢出、WebGL 崩溃**；高端机又需要 **清晰度**。同一套高精模型全量下发，**LCP 动辄数秒、甚至直接 OOM**；运行时 **同屏三角面过高** 导致 **FPS 个位数、长卡顿**。重材质还在主线程解析时，**和 WebGL、框架抢主线程**，交互卡死。 |
| **任务（T）** | 在 **能打开、能看清** 和 **能流畅交互** 之间做平衡，而不是「一刀切降画质」。 |
| **困难点（核心矛盾）** | **① 设备分级**：iOS **拿不到 `deviceMemory`**，Android 机型 **UA 维护成本极高**。**②** 只分级不够，还要 **按视距** 换精度。**③ 材质**：体积大、解码重，必须 **搬出主线程**。 |
| **行动（A）** | **混合分级**：Android/PC 用 **`deviceMemory` + `hardwareConcurrency` 阈值**；iOS 用 **UA 解析机型代际** 映射高/中/低。**LOD**：`THREE.LOD` + 每帧 `lod.update(camera)`，**按距离 + 分级** 切换不同精度网格。**Worker**：在 Worker 里 **fetch → `createImageBitmap`**，用 **Transferable** 把 `ImageBitmap` 传回主线程再建 `THREE.Texture`，减轻主线程阻塞。 |
| **结果（R）** | 文档结论：**场景平均 FPS 提升约 35%**（以你线上数据为准）；从指标上同时缓解 **LCP/Crash（能不能看）** 与 **FPS（流不流畅）**。 |

**一句话收口：** 「最难的是 **Web 没有统一性能基线**：用 **混合特征做分级**，再用 **LOD 解决算力**、用 **Worker 解决加载/解码占主线程**，把 **加载** 和 **渲染** 两条线拆开优化。」
